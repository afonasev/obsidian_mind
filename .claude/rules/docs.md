## Общие правила по написанию документации

- Минимальный диф: трогаем только релевантные разделы, не переписываем целиком.
- Если из изменения не очевидно, как обновить документ — спросить у разработчика, не выдумывать.

## README как источник правды

`README.md` описывает **текущее** состояние проекта верхнего уровня: возможности, стек, требования, быстрый старт, структуру, статус. Он должен совпадать с кодом, а не отставать от него.

`openspec/` — про **планы и спецификации изменений**. Содержимое в README не дублируем.

### Когда обязательно править README

Если изменение попадает под один из триггеров — апдейт README идёт в том же коммите/PR, что и сам код:

- **Новая пользовательская возможность** (новый компонент в `src/components/`, новый слой стора, новый use-case в `src/domain/`) → раздел про статус MVP (что уже работает).
- **Новая или удалённая runtime-зависимость в `package.json`**, влияющая на стек (React, `@xyflow/react`, `idb`, `zustand`, Tauri, и т. п.) → раздел «Стек».
- **Изменение слоёв архитектуры** (новый слой, переименование, новые правила зависимостей между слоями) → раздел «Структура проекта».
- **Новая команда в `Makefile`** или изменение поведения существующей → раздел «Быстрый старт».
- **Новое требование к окружению** (минимальная версия `bun`, переход на новый Rust toolchain, новые системные библиотеки) → раздел «Требования».
- **Появление новой папки верхнего уровня** или удаление существующей → раздел «Структура проекта».

### Когда НЕ нужно править README

- Багфиксы без изменения поведения для пользователя.
- Внутренний рефакторинг в пределах одного слоя без новых публичных интерфейсов.
- Изменения в `tests/`, `openspec/`, `.claude/` сами по себе.
- Косметика UI (стили, тексты) без новых фич.

### Как править

- Не дублируем CLAUDE.md и правила из `.claude/rules/` — README про *что* и *как пользоваться*, CLAUDE.md и rules про *как разрабатывать*.

## Каталог `docs/` — техническая документация

`docs/` — расширенное описание проекта для разработчиков и LLM-агентов: архитектура, фронтенд, схема хранилища, ключевые технические решения. Документы должны совпадать с кодом.

Состав каталога:
- `docs/README.md` — индекс.
- `docs/architecture.md` — слои фронта (`domain → persistence → store → components`), поток данных, роль Tauri.
- `docs/frontend.md` — UI-компоненты, стор, стили, интеграция с `@xyflow/react`.
- `docs/storage.md` — схема IndexedDB, формат `Node` / `Edge`, дебаунс автосохранения, sanitize.
- `docs/decisions/` — директория с ключевыми техническими решениями и их обоснованием. Каждый файл должен в названии иметь дату решения и краткий slug, например `2026-05-27_initial-stack.md`.

### Обязательные триггеры для апдейта `docs/`

Изменения ниже идут в том же PR, что и код. Это важнее README, потому что `docs/` — это «контракт» для будущих агентов и контрибьюторов.

| Изменение в коде                                                                                                  | Куда писать                                                  |
| ----------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| Новая запись / object store в IndexedDB, изменение формата `Node` / `Edge`, новая ветка миграции схемы            | `docs/storage.md`                                            |
| Новая Tauri-команда, изменение IPC-контракта, новая capability                                                    | `docs/architecture.md` (раздел про Tauri / IPC)              |
| Новый компонент в `src/components/`, новая глобальная утилита, изменение пропсов публичного компонента            | `docs/frontend.md`                                           |
| Новый слой / новая папка в `src/`, изменение правил зависимостей между слоями, новый паттерн DI                   | `docs/architecture.md`                                       |
| Сознательное архитектурное решение (выбор библиотеки, отказ от async, новая стратегия рендера и т. п.)            | `docs/decisions/` — добавить новый файл с **что/зачем/цена** |
| Новая обязательная переменная окружения или новое требование к окружению                                          | `docs/architecture.md` и `README.md`                         |
| Новая команда в `Makefile`                                                                                        | `README.md` (быстрый старт); в `docs/` — только если меняет поток данных |

### Когда `docs/` править НЕ нужно

- Чисто косметические правки UI (стили, опечатки в текстах).
- Рефакторинг без изменения публичного контракта (имена внутренних функций, разбиение модулей внутри одного слоя).
- Правки тестов, скриптов, спецификаций в `openspec/`.
- Багфиксы, не меняющие поведение, описанное в документации.

### Как править `docs/`

- Минимальный диф: правим только релевантные разделы, не переписываем файлы целиком.
- `docs/decisions/` — append-only по смыслу: старые решения не удаляем, помечаем устаревшими и создаём новые с обоснованием смены.
- Не дублировать содержимое между документами: ссылка лучше копии. Например, схема IDB живёт в `storage.md`, а `frontend.md` ссылается на неё.
- Перед коммитом — пробежаться по затронутому файлу глазами: не появилось ли расхождений с соседними разделами.
