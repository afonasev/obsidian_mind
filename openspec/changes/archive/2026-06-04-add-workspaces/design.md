## Context

До этой change приложение оперирует ровно одним графом mindmap: zustand-стор `mindMapStore` держит `graph` + UI-состояние, а persistence хранит граф под фиксированным ключом `current` в object store `graph` (IDB `mindmap` v1). Пространств нет — все корни живут в общем графе.

Change вводит слой пространств поверх существующих слоёв (`domain → persistence → store → components`), не ломая принцип однонаправленных зависимостей. Граф перестаёт быть глобальным синглтоном и становится принадлежащим конкретному пространству; вокруг появляется реестр пространств и активное пространство.

Ключевые ограничения (из обсуждения, зафиксированы как решения):
- старый граф **не мигрируется** — стартуем с пустого списка;
- при удалении активного пространства активным становится **соседнее**;
- при старте открывается **последнее активное** пространство;
- undo/redo ведётся **отдельно на каждое пространство** в рамках сессии (не персистится);
- создание корней запрещено, **пока нет активного пространства**.

## Goals / Non-Goals

**Goals:**
- Несколько независимых именованных пространств, каждое со своим графом.
- Сворачиваемая панель слева: список, `[+]`, `⋮`-меню (rename/delete с подтверждением).
- Переключение активного пространства меняет видимый граф канваса.
- Привязка корней к пространству; пустое состояние, когда пространств нет.
- Сохранение активного пространства и состояния панели между перезапусками.
- Отдельная undo/redo-история на пространство (в памяти сессии).

**Non-Goals:**
- Перетаскивание/ручная сортировка пространств в списке (порядок = порядок создания).
- Миграция ранее сохранённого графа `current`.
- Персист undo/redo между перезапусками.
- Перенос узлов/поддеревьев между пространствами.
- Вложенность пространств, иконки/цвета, поиск по пространствам.

## Decisions

### Решение 1. Схема IndexedDB v2: три object store

Поднимаем `DB_VERSION` до `2`. Структура:

- `workspaces` — key = `workspaceId` (string), value = `{ id, name, createdAt }`. Порядок в списке — сортировка по `createdAt`.
- `graph` — key = `workspaceId` (вместо фиксированного `current`), value = `{ version: 2, nodes, edges, updatedAt }`. При удалении пространства запись графа удаляется по тому же ключу.
- `meta` — key = строковая константа, value под ключами `activeWorkspaceId: string | null` и `panelCollapsed: boolean`.

**Миграция v1→v2**: в `upgrade` старый object store `graph` пересоздаётся (старая запись `current` отбрасывается), добавляются `workspaces` и `meta`. Данные v1 не переносятся — осознанное решение, у реальных пользователей графа пока нет.

*Альтернатива (отклонена)*: один store с составными записями `{workspaces, graphs, meta}` под одним ключом — проще миграция, но любая мутация графа переписывает весь блоб, теряем гранулярность дебаунса и плохо масштабируется по числу пространств.

### Решение 2. Один стор; активный срез = граф активного пространства

Остаёмся с единственным `mindMapStore`. Видимый срез (`graph / selectedNodeId / editingNodeId / past / future`) всегда относится к **активному** пространству. Добавляются поля состояния:
- `workspaces: readonly Workspace[]`
- `activeWorkspaceId: NodeId | null`
- `panelCollapsed: boolean`

Истории неактивных пространств хранятся вне zustand-состояния, в замыкании `createMindMapStore`: `Map<workspaceId, { past, future }>` (как `coalesceKey`/`clipboard` сейчас) — менять их не должно вызывать ререндер.

**Переключение A→B** (`selectWorkspace`):
1. синхронный `flush` дебаунс-сейвера — иначе отложенная запись графа A уйдёт под ключ B;
2. стэшим текущие `past/future` в `Map[A]`;
3. грузим граф B из IDB по ключу B;
4. восстанавливаем `past/future` из `Map[B]` (пусто, если в этой сессии B не открывали);
5. сбрасываем `selectedNodeId/editingNodeId` и transient-бухгалтерию (`clearTransient`);
6. пишем `activeWorkspaceId = B` в `meta`.

*Альтернатива (отклонена)*: отдельный стор на каждое пространство — дублирует подписки/сейверы, усложняет общий тулбар undo/redo, не оправдано.

### Решение 3. Новый домен `workspaces` — чистые операции

`src/domain/workspaces.ts`: тип `Workspace { readonly id, readonly name, readonly createdAt }` и чистые функции над списком: `createWorkspace(list, {id, name, createdAt})`, `renameWorkspace(list, id, name)`, `removeWorkspace(list, id)`, плюс `neighborOf(list, id)` — выбор соседнего пространства после удаления (следующее по списку, иначе предыдущее, иначе `null`). Без импортов React / IDB. Граф остаётся отдельным доменом (`domain/graph.ts`) — пространство хранит только метаданные, не граф.

### Решение 4. Поток сохранения: граф — дебаунсом, метаданные — немедленно

- **Граф** активного пространства сохраняется дебаунс-сейвером по ключу `activeWorkspaceId`. `bindSaver` при изменении `state.graph` планирует запись для текущего активного id; смена активного пространства предваряется `flush` (см. Решение 2).
- **Реестр пространств и meta** (`activeWorkspaceId`, `panelCollapsed`) малы и редко меняются — сохраняем немедленно при каждой операции (create/rename/delete/select/toggle), без дебаунса.

### Решение 5. Создание через inline-rename; пустое имя → дефолт

`createWorkspace` (экшен стора): генерирует `id`, добавляет пространство с временным/пустым именем, делает его активным и **сразу включает режим редактирования имени** в панели (по аналогии с редактированием узла). На завершении ввода:
- непустое имя — сохраняется как есть;
- **пустое имя → подставляется дефолт** `«Новое пространство»` (чтобы не остаться без пространства после `[+]`).
Переименование существующего в пустое — отклоняется, остаётся прежнее имя.

### Решение 6. Пустое состояние = нет пространств

Так как активное пространство выбирается всегда (старт → последнее активное; удаление → сосед), состояние «ничего не выбрано» возникает только когда список пуст (свежая установка или удалили последнее). В этом случае канвас рисует подсказку «создайте пространство», `addRoot` — no-op. Отдельного режима «есть пространства, но не выбрано» не вводим.

### Решение 7. Панель — отдельный компонент `WorkspacePanel`

Новый `src/components/WorkspacePanel/`. Свёрнутое состояние читается из стора (`panelCollapsed`) и персистится в `meta`. `⋮`-меню и попап подтверждения удаления — внутри панели; доступность через `getByRole` (кнопки, диалог подтверждения). `App` рендерит панель рядом с `Canvas`.

## Risks / Trade-offs

- **Гонка дебаунса при переключении** (граф A уходит под ключ B) → синхронный `flush` перед сменой `activeWorkspaceId` (Решение 2, шаг 1).
- **Потеря данных v1 при апгрейде** → осознанно принято (у пользователей графа нет); зафиксировано как **BREAKING** в proposal.
- **Рост памяти от Map историй** при многих открытых за сессию пространствах → истории — это ссылки на иммутабельные снапшоты графа (как сейчас), и каждая ограничена `MAX_HISTORY`; суммарно приемлемо для личного инструмента.
- **Асинхронная загрузка графа при переключении** даёт краткий кадр до отрисовки → `selectWorkspace` — async, канвас отрисовывает граф после загрузки; промежуточно показываем текущий/пустой без ошибки.
- **100% coverage** (правило проекта) → новый домен и экшены стора покрываются unit-тестами честно (`fake-indexeddb`), панель — `@testing-library/react` + `user-event`, ключевой сценарий переключения — e2e.
