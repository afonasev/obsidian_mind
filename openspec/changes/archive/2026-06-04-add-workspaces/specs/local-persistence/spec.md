## MODIFIED Requirements

### Requirement: Хранение графа в IndexedDB

Приложение SHALL хранить графы mindmap (узлы и рёбра) в локальной базе IndexedDB браузера / webview, без зависимости от внешнего сервера. База SHALL иметь имя `mindmap` и версию `2`. Внутри базы SHALL существовать object store `graph`, где граф каждого пространства хранится отдельной записью под ключом `workspaceId`; object store `workspaces` со списком пространств; object store `meta` для активного пространства и UI-настроек.

#### Scenario: Первый запуск создаёт пустую базу

- **WHEN** приложение запускается впервые на устройстве (база `mindmap` не существует)
- **THEN** создаётся база `mindmap` версии `2`, object stores `graph`, `workspaces` и `meta`, записей графов и пространств нет, канвас показывает подсказку создать пространство

#### Scenario: Структура хранимой записи графа

- **WHEN** в store `graph` присутствует запись под ключом `workspaceId`
- **THEN** значение записи имеет форму `{ version: 2, nodes: Node[], edges: Edge[], updatedAt: number }`, где `Node` содержит `{ id, text, position: { x, y }, parentId }`, а `Edge` — `{ id, source, target }`

#### Scenario: Структура записи пространства

- **WHEN** в store `workspaces` присутствует запись пространства
- **THEN** значение записи имеет форму `{ id: string, name: string, createdAt: number }`

### Requirement: Автозагрузка графа при старте

Приложение SHALL автоматически загружать граф активного пространства из IndexedDB при старте и отрисовывать его на канвасе. Активное пространство SHALL определяться по сохранённому в `meta` значению.

#### Scenario: Граф активного пространства восстанавливается при перезагрузке

- **WHEN** пользователь создал N узлов и M рёбер в активном пространстве, закрыл приложение, открыл снова
- **THEN** при старте это пространство снова активно, на канвасе отображаются те же N узлов с прежним текстом и позициями, те же M рёбер, выделение и зум сбрасываются в значения по умолчанию

#### Scenario: Старт без активного пространства

- **WHEN** при старте в `meta` нет активного пространства или список пространств пуст
- **THEN** канвас отображает подсказку создать пространство, никакая ошибка пользователю не показывается

### Requirement: Автосохранение мутаций с дебаунсом

Приложение SHALL автоматически сохранять граф активного пространства в IndexedDB после каждой мутации (создание / удаление / редактирование / перемещение узла, создание / удаление ребра). Сохранение MUST идти с дебаунсом 250 мс под ключом активного пространства. Перед переключением активного пространства приложение MUST синхронно сбросить (flush) отложенную запись, чтобы граф не был записан под чужим ключом.

#### Scenario: Одиночная мутация сохраняется в течение 250 мс

- **WHEN** пользователь редактирует текст узла и подтверждает Enter, после чего никаких других мутаций не следует
- **THEN** через ≤ 300 мс в store `graph` запись активного пространства обновлена, поле `updatedAt` содержит свежий timestamp

#### Scenario: Серия мутаций сливается в одну запись

- **WHEN** пользователь перетаскивает узел в течение 1 секунды, во время чего позиция меняется десятки раз
- **THEN** в IndexedDB происходит ровно одна запись после окончания перетаскивания (плюс возможные промежуточные, но не более одной на 250 мс окна)

#### Scenario: Переключение пространства сбрасывает отложенную запись

- **WHEN** пользователь делает мутацию в пространстве A и сразу переключается на пространство B
- **THEN** перед сменой активного пространства граф A синхронно записывается под ключом A, и никакая отложенная запись не попадает под ключ B

#### Scenario: Закрытие приложения во время дебаунса

- **WHEN** пользователь делает мутацию и сразу же закрывает вкладку / окно
- **THEN** перед закрытием приложение делает синхронный flush — последнее состояние записывается в IndexedDB до выгрузки страницы

### Requirement: Версионирование схемы хранилища

Записи графов в IndexedDB SHALL содержать поле `version: number` для возможности будущих миграций. В этой версии change значение SHALL быть равно `2`. При апгрейде с версии `1` старые данные графа (запись `current`) SHALL быть отброшены без миграции.

#### Scenario: Запись несёт версию схемы

- **WHEN** приложение сохраняет граф в первый раз
- **THEN** в записи графа поле `version` равно `2`

#### Scenario: Апгрейд с версии 1 отбрасывает старый граф

- **WHEN** на устройстве была база `mindmap` версии `1` с записью `current` и приложение запускается с новой схемой
- **THEN** база апгрейдится до версии `2`, прежняя запись `current` не переносится, список пространств пуст
