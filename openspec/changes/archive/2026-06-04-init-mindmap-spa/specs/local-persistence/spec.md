## ADDED Requirements

### Requirement: Хранение графа в IndexedDB

Приложение SHALL хранить весь граф mindmap (узлы и рёбра) в локальной базе IndexedDB браузера / webview, без зависимости от внешнего сервера. База SHALL иметь имя `mindmap` и версию `1`. Внутри базы SHALL существовать object store `graph` с одной фиксированной записью под ключом `current`.

#### Scenario: Первый запуск создаёт пустую базу

- **WHEN** приложение запускается впервые на устройстве (база `mindmap` не существует)
- **THEN** создаётся база `mindmap` версии `1`, object store `graph`, запись под ключом `current` отсутствует, на канвасе нет узлов

#### Scenario: Структура хранимой записи

- **WHEN** в хранилище присутствует запись `current`
- **THEN** значение записи имеет форму `{ version: 1, nodes: Node[], edges: Edge[], updatedAt: number }`, где `Node` содержит `{ id, text, position: { x, y }, parentId }`, а `Edge` — `{ id, source, target }`

### Requirement: Автозагрузка графа при старте

Приложение SHALL автоматически загружать сохранённый граф из IndexedDB при старте и отрисовывать его на канвасе.

#### Scenario: Граф восстанавливается при перезагрузке приложения

- **WHEN** пользователь создал N узлов и M рёбер, закрыл приложение, открыл снова
- **THEN** на канвасе отображаются те же N узлов с прежним текстом и позициями, те же M рёбер, выделение и зум сбрасываются в значения по умолчанию

#### Scenario: Загрузка из пустой базы

- **WHEN** при старте запись `current` отсутствует или содержит пустые массивы
- **THEN** канвас отображается пустым, приложение готово к созданию первого узла, никакая ошибка пользователю не показывается

### Requirement: Автосохранение мутаций с дебаунсом

Приложение SHALL автоматически сохранять граф в IndexedDB после каждой мутации (создание / удаление / редактирование / перемещение узла, создание / удаление ребра). Сохранение MUST идти с дебаунсом 250 мс, чтобы серия близких мутаций (например, протяжка узла) приводила к одной записи.

#### Scenario: Одиночная мутация сохраняется в течение 250 мс

- **WHEN** пользователь редактирует текст узла и подтверждает Enter, после чего никаких других мутаций не следует
- **THEN** через ≤ 300 мс в IndexedDB запись `current` обновлена, поле `updatedAt` содержит свежий timestamp

#### Scenario: Серия мутаций сливается в одну запись

- **WHEN** пользователь перетаскивает узел в течение 1 секунды, во время чего позиция меняется десятки раз
- **THEN** в IndexedDB происходит ровно одна запись после окончания перетаскивания (плюс возможные промежуточные, но не более одной на 250 мс окна)

#### Scenario: Закрытие приложения во время дебаунса

- **WHEN** пользователь делает мутацию и сразу же закрывает вкладку / окно
- **THEN** перед закрытием приложение делает синхронный flush — последнее состояние записывается в IndexedDB до выгрузки страницы

### Requirement: Целостность ссылок узлов и рёбер

При сохранении и загрузке приложение MUST гарантировать, что каждое ребро ссылается на существующие узлы. Если при загрузке обнаруживается «висячее» ребро (ссылка на несуществующий узел) — оно SHALL быть отброшено молча.

#### Scenario: Удалённый узел не оставляет висячих рёбер

- **WHEN** пользователь удаляет узел, имеющий рёбра
- **THEN** в IndexedDB записывается граф без удалённого узла И без рёбер, инцидентных удалённому узлу

#### Scenario: Загрузка повреждённого графа

- **WHEN** в IndexedDB лежит запись с ребром, чьи `source` или `target` указывают на отсутствующий в `nodes` id
- **THEN** при загрузке такое ребро исключается, оставшийся валидный подграф отображается без ошибки пользователю

### Requirement: Версионирование схемы хранилища

Запись в IndexedDB SHALL содержать поле `version: number` для возможности будущих миграций. В этой версии change значение SHALL быть равно `1`.

#### Scenario: Запись несёт версию схемы

- **WHEN** приложение сохраняет граф в первый раз
- **THEN** в записи `current` поле `version` равно `1`
