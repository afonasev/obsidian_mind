## Context

Узел графа (`MindNode`) сейчас несёт только `text` — короткое имя, редактируемое инлайн в `CloudNode` и показываемое в левой панели `WorkspacePanel`. Архитектура слоёв строгая: `domain → persistence → store → components`. Автосохранение уже устроено как дебаунс 250 мс на смену ссылки `graph` (`createDebouncedSaver`), с flush при переключении пространства (`leaveActiveWorkspace`) и на `beforeunload`/`pagehide` (`bindUnloadFlush`). Undo/redo — стек снимков графа; правки имени коалесцируются по ключу `text:${nodeId}` через `commit`.

Левая панель — готовый паттерн сворачиваемой панели: `panelCollapsed` в сторе, `togglePanel`, персист в `meta` под `META_PANEL_COLLAPSED_KEY`, узкая полоса с кнопкой в свёрнутом виде. Правую панель строим зеркально.

## Goals / Non-Goals

**Goals:**

- Хранить и редактировать markdown-тело узла в правой сворачиваемой панели.
- Просмотр тела как отрендеренного markdown (CommonMark + GFM), клик → правка сырого текста.
- Заголовок в панели = имя узла, двусторонняя связь с инлайн-правкой на канвасе.
- Автосохранение тела без кнопки: blur / смена узла / 1 с без ввода.
- Правка тела участвует в undo/redo.

**Non-Goals:**

- Рендер тела на канвасе (тело живёт только в панели; на layout не влияет).
- Подсветка синтаксиса в код-блоках, картинки, произвольный HTML внутри markdown (`rehype-raw` не подключаем).
- Полноценный WYSIWYG / live-preview бок о бок. Два явных режима: просмотр и правка.
- Совместное редактирование, версии тела, экспорт в `.md`-файлы.

## Decisions

### Решение 1: `body` — опциональное поле `MindNode`, без bump версии схемы

`MindNode` дополняется `body?: string`. `repository.toGraph` кастит `record.nodes` как `unknown → MindNode[]` без поля-за-полем валидации, поэтому старые записи без `body` читаются как `undefined` — миграция и bump `DB_VERSION` (остаётся `2`) не нужны. `sanitize` трогает только рёбра — тело не затрагивает.

Альтернатива (отдельный object store `bodies` по `nodeId`): отвергнута — тело логически часть узла, отдельный store ломает атомарность сохранения графа, дублирует жизненный цикл (удаление поддерева, copy/paste, undo) и усложняет `extractSubtree`/`pasteSubtree`. Поле на узле едет вместе с узлом бесплатно.

Следствие: `pasteSubtree`/`extractSubtree` копируют тело автоматически (берут весь `node`); специально ничего не добавляем, но это покрываем тестом.

### Решение 2: `updateBody` в домене — чистая, без `layout()`

`updateBody(graph, { nodeId, body })` по образцу `updateText`, но **без** вызова `layout()` в сторе: тело не отображается на канвасе и не влияет на ширину/раскладку узла. Это и проще, и дешевле (нет ребилда дерева на каждый коммит тела).

### Решение 3: коммит тела через `commit(next, \`body:${nodeId}\`)` — undo как у имени

Экшен стора `updateBody(nodeId, body)` вызывает `commit(updateBody(graph,…), \`body:${nodeId}\`)`. Коалесцирование по ключу `body:${nodeId}` collapses серию правок одного тела в один шаг undo, отдельный от правок имени (`text:${nodeId}`) и перемещений (`move:${nodeId}`). Полностью переиспользует существующий механизм истории.

### Решение 4: правка тела — локальный буфер + триггерный коммит

В режиме правки `textarea` держит **локальное** состояние (`useState`), а не пишет в стор на каждый keystroke. Коммит в стор (`updateBody`) происходит на трёх триггерах:

- `onBlur` текстового поля;
- размонтирование редактора при смене `selectedNodeId` (через cleanup `useEffect` с актуальным буфером в `ref`);
- 1 секунда без ввода (таймер сбрасывается на каждый keystroke).

Почему буфер, а не «живой» апдгрейд стора как у имени: (1) ровно описанная пользователем механика автосейва по триггерам; (2) тело — длинный текст, гонять `graph` через новую ссылку и ре-рендер `Canvas` на каждую букву — лишняя нагрузка. Имя узла остаётся «живым» (там важно, что узел растёт по ширине в реальном времени).

Заголовок (имя узла) — наоборот, «живой»: `TitleInput` пишет через `updateText` сразу, чтобы канвас и левая панель обновлялись синхронно. Это переиспользует существующий путь `updateText` (с `layout`).

### Решение 5: рендер markdown — `react-markdown` + `remark-gfm`

`react-markdown` v10 рендерит markdown в React-элементы **без** `dangerouslySetInnerHTML` и без сырого HTML по умолчанию — удовлетворяет запрет из `react.md` из коробки. `remark-gfm` добавляет таблицы, чек-листы, `~~зачёркнутое~~`, автоссылки.

Альтернативы: `marked + DOMPurify + dangerouslySetInnerHTML` (нарушает `react.md`, нужен XSS-санитайзер) — отвергнут; свой мини-рендерер — отвергнут (YAGNI наизнанку: edge-cases markdown бесконечны). Цена `react-markdown` — несколько транзитивных пакетов (`remark`/`unified`); приемлемо для desktop-приложения без бандл-бюджета. ESM-only — стек уже ESM (Vite, `idb`), проблем нет.

### Решение 6: `EditorPanel` — зеркало `WorkspacePanel`, раскладка в три колонки

Новое состояние стора: `editorCollapsed: boolean`, экшен `toggleEditor()`, персист в `meta` под новым ключом `META_EDITOR_COLLAPSED_KEY = "editorPanelCollapsed"` (по умолчанию `false` = развёрнута). В `repository` — `loadEditorCollapsed`/`saveEditorCollapsed` по образцу `loadPanelCollapsed`/`savePanelCollapsed`; загрузка в `loadWorkspaces` (добавить в `Promise.all`).

`App.tsx`: `<WorkspacePanel/> <Canvas/> <EditorPanel/>` в CSS-grid `лево | центр | право`.

Содержимое панели читает `selectedNodeId` и соответствующий узел из `graph`. «Имя родителя» — по `node.parentId`: для не-корня кнопка с именем родителя, клик вызывает `selectNode(parentId)` + `revealNode(parentId)`; для корня (`parentId === null`) строка не рендерится.

## Risks / Trade-offs

- **Коммит-при-размонтировании теряет правку, если buffer-ref не синхронен** → держим актуальный текст в `ref`, обновляемый в `onChange`; cleanup `useEffect` читает `ref.current` и коммитит только если он отличается от тела в сторе. Тест на смену узла с несохранённой правкой обязателен.
- **Двойной коммит (blur + таймер/unmount)** → коммит идемпотентен: `updateBody` с тем же текстом коалесцируется по ключу и не плодит шаги истории; перед коммитом сравниваем буфер с текущим телом и выходим, если равны.
- **`react-markdown` ESM-only под Vitest/jsdom** → проект уже на ESM, `transformIgnorePatterns` не нужен для Vite-pipeline; покрываем рендер тестом (`getByRole("heading")`, список и т.п.), чтобы поймать интеграционную поломку рано.
- **100% coverage на таймерах** → правим через `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync(1000)`, как в тестах дебаунс-сейвера; все ветки (blur / unmount / timer / пустой плейсхолдер / корень без родителя) покрываем явно.
- **GFM расширяет поверхность рендера** → ограничиваемся `remark-gfm` без `rehype-raw`; произвольный HTML не исполняется, XSS-поверхность минимальна.
