## Context

Имя узла — плоская строка `MindNode.text`, потребляемая канвасом (`CloudNode`), правой панелью (`EditorPanel.title`) и списком корней (`WorkspacePanel`). Инлайн-правка живёт в `CloudNode.EditView` (textarea, текст течёт через `updateText` → layout). Undo/redo — снапшоты графа с коалесцированием по `coalesceKey` (`mindmap-store.ts`). Layout — tidy-tree, реагирует на смену топологии и размеров узла. Прецедент опционального поля без миграции — `body?` (`updateBody`, который намеренно **не** триггерит layout, т.к. тело не на канвасе).

Эта фича добавляет к имени узла стиль на всю ноду (жирность/курсив/размер) и инлайн-тулбар. В отличие от `body`, размер шрифта **меняет габариты узла** и потому обязан триггерить layout.

## Goals / Non-Goals

**Goals:**
- Опциональный стиль имени на всю ноду: `bold`, `italic`, `fontScale` (целочисленный шаг).
- Плавающий тулбар форматирования над узлом только в режиме инлайн-правки.
- Корректный пересчёт раскладки при смене размера шрифта.
- Каждое изменение стиля — отдельный шаг undo/redo.
- Никакой миграции: старые узлы читаются как «без стиля».

**Non-Goals:**
- Форматирование выделенного куска имени (range/rich-text) — отвергнуто в explore.
- Стиль имени в правой панели и списке корней — там плоский ярлык.
- Цвет/шрифтовая семья/подчёркивание — вне MVP (YAGNI).
- Стиль для markdown-тела — у тела свой редактор.

## Decisions

### Модель данных: вложенный опциональный объект `style`

`MindNode.style?: NodeNameStyle`, где `NodeNameStyle = { bold?: boolean; italic?: boolean; fontScale?: number }`. Все поля опциональны, отсутствие = дефолт. Альтернатива — плоские поля прямо в `MindNode` (`bold?`, `italic?`, `fontScale?`) — отвергнута: вложенный объект группирует связанные данные и не засоряет корень узла. По образцу `body?` — `sanitize`/persistence не трогаем, читается как `undefined`.

### `fontScale` — относительный шаг, не пиксели

Хранится целым шагом в диапазоне (напр. `-2…+3`, база `0`). В CSS маппится на размер через переменную/класс, а не на абсолютные px в данных. Почему: дружит с темами (`add-dark-theme`), не зашивает пиксели в персистентные данные, тривиально клампится на границах. A−/A+ меняют шаг на ±1 с клампом.

### Доменная операция `updateNodeStyle`

Новая чистая функция в `graph.ts` по образцу `updateText`: принимает `nodeId` и частичный патч стиля (или итоговый стиль), возвращает новый `Graph`. Клампинг `fontScale` — в домене, чтобы инвариант диапазона держался в одном месте. Экшен стора `setNodeStyle`/тогглы оборачивают её через `applyGraphMutation`.

### Layout запускается при смене размера, но не при bold/italic

`fontScale` меняет габариты → после мутации нужен тот же пересчёт раскладки, что и у `updateText`. `bold`/`italic` габариты почти не меняют, но измерение узла в React Flow всё равно реагирует на контент; чтобы не плодить ветки, прогоняем стандартный re-layout на любое изменение стиля (это дёшево и согласуется с `updateText`). Решение: `setNodeStyle` в сторе идёт по той же ветке, что `updateText` (мутация + re-flow), а не по «тихой» ветке `updateBody`.

### Тулбар — `NodeToolbar` из `@xyflow/react`

Используем штатный `<NodeToolbar position={Position.Top} isVisible={isEditing}>` внутри `CloudNode`. Почему: он сам держит панель над узлом в координатах графа, корректно ведёт себя при зуме/панораме, без ручного позиционирования. Кнопки — `onMouseDown` с `preventDefault`, чтобы клик не уводил фокус из textarea и не коммитил правку (`onBlur=commit`). Это ключевой нюанс: тулбар не должен закрывать режим правки.

### Undo/redo: каждый шаг стиля отдельный

Изменение стиля проходит через `applyGraphMutation` с собственным `actionKey` (напр. `style:<nodeId>`), без коалесцирования с правкой текста и перемещением. Серию A+ A+ не коалесцируем — клики дискретны и редки.

### Рендер стиля и в просмотре, и в правке

`TextView` и `EditView` оба применяют стиль (класс bold/italic + инлайн `fontSize` от `fontScale`, т.к. значение динамическое — допустимо инлайном по `react.md`). Иначе после выхода из правки форматирование «пропадёт». `EditorPanel.title` и `WorkspacePanel` не меняем — там стиль не применяется.

## Risks / Trade-offs

- **Клик по тулбару коммитит правку (теряется режим)** → кнопки используют `onMouseDown`+`preventDefault`, не `onClick`; покрываем сценарием «клик по кнопке не выходит из правки».
- **Размер шрифта не пересчитал раскладку → соседи наезжают** → `setNodeStyle` обязательно идёт по layout-ветке; покрываем сценарием раскладки.
- **Асимметрия: на канвасе жирно, в списке корней — нет** → осознанное решение MVP (тулбар только на канвасе); зафиксировано в спеке отдельным сценарием.
- **Старые узлы без `style`** → все обращения через опциональный доступ с дефолтами; покрываем сценарием «узел без стиля».
- **100% coverage** → новые ветки (кламп на границах, тогглы, undo) требуют тестов домена/стора/компонента; учтено в задачах.
