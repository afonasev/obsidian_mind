## Context

В сторе уже есть две стопки `past` / `future` — это **undo/redo снимков графа**
(структурные изменения: создание, удаление, перемещение узлов). Выделение узла
живёт в `selectedNodeId`, переключение пространства — в асинхронном
`selectWorkspace` → `enterWorkspace` (грузит граф из IndexedDB, сбрасывает
`selectedNodeId` в `null`). Гистории неактивных пространств для undo хранятся в
closure-`Map` стора и не персистятся.

Фича добавляет **третью, ортогональную ось** — историю фокуса: куда смотрел
пользователь. Она не трогает граф, только перемещает выделение (и при
необходимости активное пространство).

## Goals / Non-Goals

**Goals:**
- Браузер-подобная навигация «Назад/Вперёд» по последовательности
  сфокусированных узлов, в том числе через границы пространств.
- Поведение, точно совпадающее со сценарием пользователя: каждый клик/стрелка —
  отдельный шаг; «Назад» откатывает по одному; «Вперёд» возвращает.
- Минимальный диф и чёткое разделение с undo/redo.

**Non-Goals:**
- Персист истории между сессиями (только память).
- Запись в историю выбора, возникшего как побочный эффект создания/вставки/
  удаления узла (это «изменения», а не «навигация»).
- Запись переключения пространства без выбора узла как отдельной точки.

## Decisions

### 1. Модель: линейный список + курсор

```
NavEntry = { workspaceId: string, nodeId: NodeId }
navHistory: readonly NavEntry[]   // полная лента
navCursor:  number                // индекс текущей позиции; -1 = пусто
```

`canGoBack = navCursor > 0`, `canGoForward = navCursor < navHistory.length - 1`.
Глубина ограничена тем же `MAX_HISTORY = 100`, что и undo (сброс старейших с
головы). Альтернатива — две стопки back/future, как у undo — отвергнута: курсор
по одному массиву проще для «сверки видимого состояния» (см. п.4).

### 2. Точка записи — `selectNode`

Запись добавляется только в `selectNode(id)` при `id !== null`. Это
единственный путь для кликов (`handleNodeClick`) и стрелок (`navigate`).
Создание/вставка/реparent ставят `selectedNodeId` напрямую, мимо `selectNode`, —
и в историю осознанно не попадают. Дедуп: если `(activeWorkspaceId, id)` уже
равно `navHistory[navCursor]`, запись не добавляется. Новый выбор обрубает всё
после курсора (forward-хвост) и пушит в конец — ветвление, как у undo.

### 3. Переход — асинхронный, под флагом `navigating`

`goBack` / `goForward` — `async`: если целевая запись в другом пространстве,
вызывается существующий `selectWorkspace(workspaceId)` (грузит граф), затем
`selectNode(nodeId)`. На время перехода поднимается closure-флаг `navigating`,
который глушит запись в `selectNode` — иначе переход зациклит историю. Флаг —
вне zustand-состояния (не должен вызывать ре-рендер), как `coalesceKey`.

### 4. Правило сверки «видимое состояние ↔ курсор»

Свитч пространства и снятие выделения в историю не пишутся, поэтому видимое
состояние `(activeWorkspaceId, selectedNodeId)` может разойтись с
`navHistory[navCursor]`. Чтобы первый «Назад» возвращал к последнему
сфокусированному узлу, а не перепрыгивал через него:

```
target(back)    = matchesCursor() ? navCursor - 1 : navCursor
target(forward) = matchesCursor() ? navCursor + 1 : navCursor
matchesCursor() = navHistory[navCursor] совпадает с (activeWorkspaceId, selectedNodeId)
```

То есть при расхождении и «Назад», и «Вперёд» сперва возвращают на текущий
курсор (снап), и лишь следующий клик шагает дальше. Это и делает сценарий
«свитч в W2 без выбора → Назад → возврат к B@W1» интуитивным.

### 5. Битые записи — перешагивание

При переходе целевая запись проверяется: существует ли её пространство в
`workspaces` и (после загрузки графа) узел в графе. Если нет — двигаемся в ту же
сторону к следующей валидной записи, обновляя курсор. Если валидных в этом
направлении не осталось — курсор встаёт на крайнюю достижимую, переход
безопасно завершается. `deleteWorkspace` дополнительно вычищает записи
удалённого пространства из `navHistory` и корректирует `navCursor`
(пересчёт индекса относительно оставшихся записей).

### 6. Слои

- `src/domain/nav-history.ts` — чистые функции над `{ history, cursor }`:
  `record`, `back`, `forward`, `pruneWorkspace`. Без React/IDB/стора, тестируются
  изолированно (проверка узла в графе остаётся в сторе — домен графа не знает).
- `src/store/mindmap-store.ts` — состояние + `async goBack/goForward`, селекторы.
- `src/components/FocusNav/` — две кнопки поверх канваса (левый верх),
  `disabled` по `canGoBack/canGoForward`.
- `Canvas` keydown — Alt+← / Alt+→ → `goBack/goForward`.

## Risks / Trade-offs

- **Асинхронный переход при быстрых нажатиях** → защита: повторный `goBack` во
  время незавершённого перехода может прийти раньше загрузки графа. Митигировать
  тем же флагом `navigating` (ранний выход, если переход уже идёт), либо
  сериализацией через `await`. Решаем в реализации простейшим вариантом —
  ранний выход при поднятом флаге.
- **Расхождение курсора и видимого состояния** → закрыто правилом сверки (п.4);
  риск в том, что пользователь не сразу поймёт «первый Назад вернул туда же».
  Принимаем как стандартное браузер-подобное поведение.
- **Тестовое покрытие 100%** → асинхронные пути перехода и перешагивание битых
  записей дают много веток; закладываем юнит-тесты домена + тесты стора с
  fake-indexeddb на кросс-воркспейсные переходы.
- **Отказ от персиста** → после перезагрузки история пуста; приемлемо (как undo).
