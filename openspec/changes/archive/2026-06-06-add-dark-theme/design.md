## Context

Цвета приложения сейчас захардкожены хексами (~40 значений) в 8 CSS-модулях; единого слоя токенов нет, глобального стиля нет — `App.module.css` единственный «верхний». Канвас строится на `@xyflow/react@12.3.5`, у которого есть штатная поддержка тем (`colorMode`), горизонтальная панель `<Controls orientation>` и пользовательские кнопки `<ControlButton>`. Триггер справки (`?`) сейчас живёт в правом верхнем углу (`HotkeysHelp`), панель `<Controls>` — внизу слева.

## Goals / Non-Goals

**Goals:**
- Полноценная тёмная тема, покрывающая панели, узлы и канвас.
- Системная тема как значение по умолчанию при первом запуске; далее — выбор пользователя.
- Минималистичный UI: переключатель темы и справка собраны в горизонтальную панель `<Controls>`.
- Отсутствие вспышки светлой темы на старте.

**Non-Goals:**
- Третий режим «следовать системе вживую» (живая подписка на смену темы ОС). Система используется только как однократный посев при первом запуске.
- Кастомные пользовательские палитры / более двух тем.
- Перенос остальных плавающих элементов (`FocusNav`) — он остаётся на месте.

## Decisions

### Слой дизайн-токенов через CSS-переменные

Новый глобальный CSS-файл (например, `src/theme.css`), импортируемый в `main.tsx`: `:root { --color-bg; --color-fg; … }` для светлой темы и `[data-theme="dark"] { … }` для тёмной. Все 8 `*.module.css` переводятся с хексов на `var(--…)`.

**Почему так, а не альтернативы:** CSS Modules уже приняты в проекте (см. `react.md`), а CSS-переменные каскадируются и работают с модулями без сборочной магии. Отклонено: Tailwind/CSS-in-JS (запрещены конвенциями), отдельные «тёмные» классы на каждом компоненте (дублирование, рассинхрон).

Набор токенов выводится из текущей светлой палитры — семантические имена (фон, текст, граница, акцент, тени), а не «по компонентам». Близкие хексы схлопываются в один токен.

### Хранение и инициализация темы

Выбор хранится в `localStorage` под одним ключом (значение `light` | `dark`). На старте: `stored ?? (matchMedia("(prefers-color-scheme: dark)").matches ? "dark" : "light")`, результат пишется в `document.documentElement.dataset.theme`.

**Почему localStorage, а не IDB-стор:** тема — глобальная UI-преференция, не часть графа; localStorage синхронен и читается до первого кадра. IDB-стор асинхронен → гарантированная вспышка.

Чтобы не было вспышки, начальное проставление `data-theme` делает **инлайн-скрипт в `<head>` `index.html`** (читает localStorage/`matchMedia`, ставит атрибут до загрузки бандла). React-слой при монтировании лишь синхронизируется с уже выставленным атрибутом.

### Переключатель темы и состояние в React

Маленький хук (напр. `useTheme`) отдаёт текущую тему и `toggle`. Источник истины — `data-theme` на `<html>` + `localStorage`; для подписки React используем `useSyncExternalStore` (соответствует `react.md`, без `useEffect`-синхронизации производного состояния). `toggle` пишет в localStorage, меняет атрибут и нотифицирует подписчиков.

### Интеграция с `@xyflow/react`

- `<Controls orientation="horizontal">` — горизонтальная панель.
- Кнопки темы (`🌙`/`☀️`) и справки (`?`) — как `<ControlButton>` внутри `<Controls>`.
- `<ReactFlow colorMode={theme}>` — синхронизация канваса (рёбра, фон, штатные контролы) с активной темой.

### Перенос справки

В `HotkeysHelp` триггер-кнопка убирается из своего угла; саму кнопку отрисовывает `Canvas` внутри `<Controls>`, а состояние открытия диалога остаётся в `HotkeysHelp` (или поднимается до `Canvas`). Диалог перепривязывается к низу-слева вместо `top: 40px; right: 0`.

## Risks / Trade-offs

- **Дублирование логики выбора темы между инлайн-скриптом (`index.html`) и хуком (`useTheme`)** → держим обе реализации тривиальными и идентичными по правилу выбора; ключ localStorage и значения (`light`/`dark`) фиксируем как контракт в спеке/доках.
- **100% coverage:** инлайн-скрипт в `index.html` вне покрытия Vitest; логику выбора темы выносим в тестируемую функцию/хук в `src/`, а инлайн-скрипт держим минимальным. Если строка в скрипте непокрываема — это HTML, не TS, в coverage не входит.
- **Перенос `?` из привычного правого-верхнего угла** → осознанное решение ради минимализма (согласовано с пользователем).
- **Неполная перекраска:** легко пропустить отдельный хекс в модулях → проверяем грепом отсутствие сырых hex/`rgba(...)` в `*.module.css` после миграции (кроме осознанных исключений с WHY-комментарием).
- **Рассинхрон нашей `data-theme` и xyflow `colorMode`** → единственный источник темы (хук) питает оба; xyflow получает `colorMode` пропом из того же значения.
