## Context

Завершение перетаскивания (`handleNodeDragStop`, `Canvas.tsx:240`) сейчас бинарно: drop над узлом → `reparent`, drop над пустотой → `dropNode` (moveNode + `relayout`). `dropNode` сохраняет позицию, но `relayout` (`layout()` в `domain/layout.ts`) притягивает не-корневой узел обратно к родителю — сохранённая позиция влияет лишь на порядок братьев (сортировка по `y`) и сторону ветки (знак `x` относительно корня).

`layout()` — чистая идемпотентная функция: корни остаются на сохранённых позициях (`layout.ts:126`), потомки выводятся детерминированно. Её результат персистится: `relayout` → `commit` записывает новые позиции в стор и автосохранение. Между корнями нет проверки пересечений, поэтому их поддеревья могут накладываться.

Перетаскиваемая позиция узла во время drag известна (`handleNodeDrag`/`handleNodeDragStop` получают `node.position`), позиция родителя — из графа. Уже есть `findDropTarget` с оценкой размеров через `estimateNodeWidth/estimateNodeHeight` и обходом поддерева `subtreeIds` — те же примитивы переиспользуем.

## Goals / Non-Goals

**Goals:**
- Открепление не-корневого узла в новый корень при отпускании далеко от родителя над пустым холстом; поддерево едет следом.
- Визуальный сигнал «рвущегося» ребра, пока во время drag порог открепления пройден.
- Инвариант: поддеревья разных корней не пересекаются; раскладка разводит налезающие корни, оставаясь идемпотентной.
- Сохранить текущее поведение близкого drop и перепривязки.

**Non-Goals:**
- Горизонтальное разведение корней «бок о бок» — разрешение коллизий только по вертикали.
- «Пружинирование» корня обратно на исходное место после ухода соседа — сдвинутая позиция перезаписывается и остаётся.
- Анимация открепления/разведения. Переключатель/жест открепления отдельной кнопкой — только drag-порог.

## Decisions

### Доменная операция `detachAsRoot(graph, {nodeId, position})`
Зеркало `reparentSubtree`: валидирует (узел существует, `parentId !== null`), иначе возвращает ту же ссылку (no-op, без истории). При успехе — `parentId → null`, новая `position`, удаляет ребро с `target === nodeId`. Дети сохраняют свои `parentId`, поэтому поддерево остаётся связным, а узел становится новым корнем своей ветки.
*Альтернатива:* расширить `reparentSubtree` с `newParentId: null` — отвергнуто: смешивает два разных контракта (перепривязка vs открепление) и усложняет валидацию циклов.

### Порог «далеко» — кратность зазору
Открепление при `dist(node, parent) > estimateNodeWidth(parent) + LAYOUT_HGAP + K · LAYOUT_HGAP`. Базовая часть `parentWidth + LAYOUT_HGAP` — нормальное расстояние до слота ребёнка; спокойно лежащий ребёнок порог не проходит. `K` — на сколько зазоров дальше нормального места, стартово `K = 2`. Хелпер в `domain/layout.ts` (рядом с оценками размеров), чистый и тестируемый.
*Альтернатива:* сырой фикс в пикселях — отвергнут: ребёнок в покое уже ~250px от родителя, сырой порог ложно срабатывал бы в покое и не масштабировался с размером шрифта/узла.

### Развилка drop — третья ветка
`handleNodeDragStop`: `dropTargetId !== null` → `reparent`; иначе если узел не корень и порог пройден → `detach`; иначе → `dropNode`. Во время `handleNodeDrag`, когда цели нет и порог пройден, выставляем transient-флаг кандидата на открепление в сторе (по аналогии с `dropTargetId`), который читает рендер рёбер для «рвущегося» стиля. Флаг сбрасывается в конце drag.

### Разведение корней — финальный проход в `layout()`
После раскладки ветвей всех корней считаем bounding box каждого поддерева (объединение боксов узлов по `estimateNodeWidth/Height`). В детерминированном порядке (по `y`, затем по `id` для устойчивости) корни, чей bbox пересекает уже принятый, сдвигаются вниз на величину вертикального перекрытия + поле; сдвиг применяется ко всем позициям поддерева в map. Внутри одного прохода итерируем до стабилизации (сдвиг может задеть следующий корень).
**Идемпотентность:** сдвиг ровно до устранения пересечения ⇒ на следующем прогоне пересечения нет ⇒ нулевой сдвиг. Порядок фиксирован ⇒ результат детерминирован. Это позволяет безопасно персистить результат `layout()`.
*Альтернатива A (авто-упаковка корней):* отвергнута пользователем — убирает ручное размещение. *Альтернатива (сдвигать только активный корень):* `layout()` чистая и не знает «активный» — глобальный детерминированный проход чище.

### Синергия с откреплением
`detach` → `relayout`, и новый проход коллизий сам отодвигает свежий корень от чужих веток. Отдельной логики размещения нового корня не нужно.

## Risks / Trade-offs

- **Дрейф позиций корней при многократных relayout** → проход идемпотентен (нулевой сдвиг без пересечения), поэтому без новых пересечений позиции стабильны; покрываем тестом «повторная раскладка ничего не двигает».
- **Вертикальный сдвиг не разводит корни «бок о бок» с глубокими ветками** → осознанное упрощение MVP; типичная раскладка майнд-мапа вертикальная. Зафиксировано в Non-Goals.
- **Ложное срабатывание/несрабатывание порога** → `K` вынесен константой, тестируется на граничных расстояниях; подкручивается без структурных правок.
- **Сдвинутый корень не возвращается на исходное место** → принятый wart ради простоты и идемпотентности (Non-Goals).
- **Производительность bbox-прохода** → O(узлы) на сбор боксов + O(корни²) на сравнение пар; число корней мало, в рамках текущих масштабов незаметно.
