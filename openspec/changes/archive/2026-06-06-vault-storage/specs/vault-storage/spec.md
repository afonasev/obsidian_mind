## ADDED Requirements

### Requirement: Раскладка базы знаний на файлы vault

Приложение SHALL отображать содержимое базы знаний на структуру каталогов и файлов внутри активного vault. Папка верхнего уровня vault SHALL соответствовать пространству; подпапка пространства SHALL соответствовать корню (одному дереву); `.md`-файл внутри папки корня SHALL соответствовать узлу с непустым телом. Узел с пустым телом SHALL не иметь отдельного файла.

#### Scenario: Создание пространства и корня создаёт папки

- **WHEN** пользователь создаёт пространство, а в нём — корень
- **THEN** в корне vault появляется папка пространства, внутри неё — папка корня

#### Scenario: Узел с телом становится заметкой

- **WHEN** пользователь добавляет узлу непустое тело
- **THEN** в папке его корня появляется `.md`-файл с этим телом; узлы без тела файлов не создают

#### Scenario: Тело удалено — файл удаляется

- **WHEN** пользователь очищает тело узла, у которого был `.md`-файл
- **THEN** соответствующий `.md`-файл удаляется, а сам узел остаётся в структуре

### Requirement: Служебная вёрстка в скрытой `.mind/`

Приложение SHALL хранить вёрстку (набор узлов, дерево `parentId`, позиции, стиль, свёрнутость, позиции корней, список и порядок пространств) в скрытой папке `.mind/` в корне vault, зеркалящей структуру пространств и корней. Пользовательские заметки SHALL не содержать координат и стиля — только `id` во frontmatter и тело.

#### Scenario: Вёрстка пишется в .mind, заметки остаются чистыми

- **WHEN** пользователь двигает узлы, меняет их стиль и сворачивает ветки
- **THEN** изменения вёрстки записываются в файлы внутри `.mind/`, а `.md`-заметки содержат только `id` во frontmatter и markdown-тело

#### Scenario: Заметка несёт идентификатор узла

- **WHEN** в папке корня присутствует `.md`-файл узла с телом
- **THEN** его frontmatter содержит поле `id`, равное идентификатору узла в вёрстке корня

### Requirement: Связка заметки с узлом по идентификатору

Приложение SHALL связывать `.md`-заметку с узлом по `id` из frontmatter заметки, а не по имени файла. Переименование или перенос файла, при котором `id` во frontmatter сохранён, SHALL не разрывать связь узла с его телом.

#### Scenario: Имя файла разошлось, id совпадает

- **WHEN** имя `.md`-файла не совпадает с записанным в вёрстке, но `id` во frontmatter сходится с узлом
- **THEN** тело привязывается к этому узлу по `id`

#### Scenario: Идентификатор отсутствует

- **WHEN** у `.md`-файла нет `id` во frontmatter
- **THEN** приложение пытается связать по имени файла; если не удаётся — узел остаётся без тела, ошибка не показывается

### Requirement: Мягкое чтение с дефолтами

Приложение SHALL читать содержимое vault устойчиво к порче и ручным правкам служебных файлов. Контент (папки и `.md`-заметки) SHALL считаться источником правды о существовании сущностей; данные `.mind/` SHALL считаться восстановимой вёрсткой. Любая отсутствующая или повреждённая часть вёрстки SHALL замещаться дефолтами по-элементно, без необработанной ошибки.

#### Scenario: Повреждена запись одного узла

- **WHEN** в `root.yaml` запись одного узла не парсится
- **THEN** этот узел берёт дефолтные вёрстку/позицию (либо пропускается), остальные узлы корня читаются нормально

#### Scenario: Отсутствует root.yaml

- **WHEN** в папке корня нет читаемого `root.yaml`, но есть `.md`-заметки
- **THEN** дерево восстанавливается как плоский список: все заметки — дети корня, позиции по дефолтной раскладке

#### Scenario: Удалена вся .mind

- **WHEN** пользователь удалил папку `.mind/` целиком и открыл vault
- **THEN** ни одна заметка не теряется: пространства/корни/узлы восстанавливаются из папок и `.md`, вёрстка берётся по дефолту

### Requirement: Доступ к хранилищу через порт с адаптерами

Логика приложения SHALL обращаться к хранилищу через единый порт `VaultStore`, имеющий взаимозаменяемые реализации: файловую (поверх доступа к vault) и in-memory (для сред без файловой системы). Поведение чтения/записи SHALL быть одинаковым для обеих реализаций.

#### Scenario: Одинаковое поведение адаптеров

- **WHEN** один и тот же сценарий чтения/записи выполняется через файловый и через in-memory адаптер
- **THEN** наблюдаемая доменная модель идентична
